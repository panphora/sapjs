How sap works

A live, source-grounded explainer of the shipped engine. Every claim cites the real code in src/; every demo below runs the actual library inlined into this file. No network, no build step, no mocks.

TL;DR

The DOM is the only state store. On every change sap throws away its in-memory state and rebuilds it by reading the live DOM top to bottom, computes, paints what changed, then discards the objects. The file you save is the app.

The five answers, in one line each:

1. Transient values live only in live DOM (a runtime expando or the control's live .value) and the serializable attribute is actively stripped every pass, so they never save. 2. No rich text: a bind is textContent only by design (HTML-valued state would break "one write path / view-source is correct"); to embed a real editor, wrap its subtree in sap-ignore. 3. No plugin system: the vocabulary is frozen on purpose; the one extension point is custom Sap.formats. 4. Every way to set state converges on the same recompute pass; control-backed programmatic writes use synthetic input/change, while attribute-state writes schedule directly. 5. No modal coupling: the detail lens projects a row into any container, including a native <dialog>.

00Overview

What sap is, the mental model, the API surface, and the attribute vocabulary, sourced from the shipped engine (not the spec).

The core mental model

Most reactive tools keep two copies of your data: one in a JS store, one on the page, and spend their lives keeping the two in sync. Sap deletes the second copy. The input's value is the quantity; the list items are the rows. When anything changes, sap reads the page, recomputes, and writes back. One copy, nothing to drift (README.md:23, src/pass.js:1-3).

01Transient state


  

    short answer
    
A transient value lives only in the live DOM: a runtime-only JS expando (el._sapTransient) for declared state, or the control's live .value/.checked for a bound control. The matching serializable attribute is actively removed on every read and write, so the saved file never carries it. There are two transient carriers and both strip.

The exact mechanism

  
    
    
  • Attribute-carried transient — state="secret:transient". parseStateDecl sets transient:true (dom.js:111-115). The value is kept in el._sapTransient; readTransient seeds it from the attribute-or-default once, then calls el.removeAttribute(name) and returns the stored value (dom.js:139-148). writeTransient stores and strips the same way (dom.js:150-153). Intake routes transient decls through readTransient (pass.js:42-45). The state= declaration token itself stays, so the field keeps existing.
    • 
    
  • Control-carried transient — <input bind="q" transient>. The carrier's mirror() sees transient and removes value/checked instead of writing them (carrier.js:85-90). Intake also strips them right after reading a transient bound control (pass.js:63-66). The live property is the only store.
    • 
    
  • Password mandate. bind on a type=password without transient halts the whole app at mount (E31); type=file always halts (E32) (lint.js:139-144). A file must never carry a cleartext password.
    • 
    
  • Mirroring, via one shared serializer. mirror() delegates to serializeControlToAttributes (control-serialize.js, the single source of truth shared with hyperclayjs): text/number/checkbox/radio write their attribute, select writes selected on the chosen option, textarea writes a cursor-safe data-value (carrier.js:89-97). contenteditable/text-leaf need no mirror — they are textContent, saved as-is. mirror runs on a carrier write; a plain typed value is made durable by persist (below).
    • 
  


  

    Save-cycle scope, precisely. The shipped engine guarantees the transient value is excluded from serializable attributes (verified in test/transient-and-codes.test.js:33-41,71-77). It does not, on its own, re-derive paints that depend on a transient field at save time. So a transient search box that drives show/hidden on rows can leave those hidden attributes in the saved bytes. The snapshot re-derivation described by the spec (05-...:28,100-102) is not part of this standalone build (src/platform.js handles live-sync + undo refreshes, not save cloning). The "Show saved HTML" demo below therefore pairs a transient field with a normal one and shows nothing derived.
    

One more precise fact the demo proves: user typing updates the live .value property but does not mirror to the value attribute on its own. Carrier writes mirror (carrier.write() → mirror(), run by Sap(el).x=v, set:, reset, projection — carrier.js:112-116), and a control marked persist mirrors on every edit (next section). So a typed value is durable when it is written or marked persist; the transient field is excluded either way.


  
Durability: persist

  
sap honors persist natively — no host platform required. On every edit of a [persist] control inside a [sap] root, the engine mirrors the live value to a serializable attribute through the shared serializer (actions.js:240 → control-serialize.js). hyperclayjs honors the same attribute on the platform through that same serializer, so the two agree by construction. (lint.js also allow-lists the host region attributes no-save/no-watch/no-undo.)

  
A textarea is the one control whose value the browser does not reload from an attribute (it reloads from child text), so its cursor-safe mirror lives in data-value. sap closes that loop on load: when it mounts a root it restores any textarea[data-value] into the textarea's content and strips the attribute (mount.js → rehydrateControlFromAttributes), so a persisted textarea round-trips standalone, no save step required. On the platform the save already finalized it to textContent, so there is nothing left to restore.

  

    The save-survival footgun. bind makes a control reactive; it does not by itself make a typed value durable. When a user types, sap updates the live .value and reschedules, but a plain bound control does not mirror to the serializable attribute (only carrier writes do). Mark it persist and the engine mirrors it on every edit (actions.js:240), so the typed value survives a plain DOM save — standalone, and on the platform where hyperclayjs does the same. Rule of thumb: bind is reactivity, persist is durability. Put persist on any user-editable control whose value must survive a reload. It is matched per control, so it goes on the control itself, not a wrapper.

sap and hyperclayjs do not fight: both write the same value through the same shared serializer (control-serialize.js), so the writes are idempotent. But never combine transient + persist: their intent is opposite. transient strips the value every pass; persist forces the live value into the saved bytes and wins, leaking exactly what transient promised to drop. On the platform, hyperclayjs's [persist] also deliberately skips hidden, password, and file: a bound hidden input already persists through sap's carrier mirror, and a password must stay transient or sap halts (E31, above).

02Rich text editing


  

    short answer
    
No rich text, by design. A bind is textContent only for the text-leaf kind, and contenteditable="plaintext-only" is the supported authoring pattern. This is an architectural constraint, not a missing feature: state values are scalars that round-trip as text. HTML-valued state would break both "one write path" and "view-source is correct."

  


  
The mechanism

  
    
    
  • kindOf returns "leaf" for a contenteditable or bound text leaf (carrier.js:16). readBy leaf returns el.textContent; writeBy leaf sets el.textContent = String(v) (carrier.js:46-47, 74-76). A bound editable reads and writes plain text, never innerHTML.
    • 
    
  • Text paints write el.textContent with String() coercion (pass.js:297-303). Every carrier write coerces with String(v) (carrier.js:63-79). Values are scalars or arrays of scalars, never DOM fragments.
    • 
    
  • A bind on a container with element children halts with E20 ("a write would overwrite its children"), but a contenteditable container with children binds fine because it is treated as a leaf (lint.js:149-153, test/transient-and-codes.test.js:126-138). The editable still reads as textContent, so any inner markup is dropped from state.
    • 
  


  

    One honesty correction vs. the spec. The spec says plumbing rewrites text binds to contenteditable="plaintext-only" (02-vocabulary.md:14,45). The shipped linter does not stamp or enforce that — it accepts any contenteditable value except "false" (lint.js:149-153). So plaintext-only is the documented authoring pattern you should write; the carrier reads/writes textContent either way.


  
Why it is architectural, not lazy

  

    
1 · escaping vs. injection
If a field held HTML, painting it via textContent would show literal <b> tags, and painting via innerHTML would be an injection vector that does not deterministically round-trip.

    
2 · view-source must be correct
"The file you save is the app, correct before JS runs" breaks if a saved value does not match what renders. Scalars-as-text always match.

    
3 · one carrier, four paths
The scalar contract lets one carrier serve intake, proxy-set, undo replay, and mount-seed. An HTML-valued kind would violate the serializable-scalar invariant (05-...:46,100).

    
What you get instead
A plaintext contenteditable, native form controls, and structure expressed as real DOM (lists, show, classes). Formatting is markup, not state.

  


  
Embedding a real editor (the escape hatch)

  
"No rich text" is about engine state, not the page. You can embed a real editor — a formatting contenteditable, or Slate / TinyMCE / ProseMirror — by telling sap to leave that subtree alone. The seam is sap-ignore.

  
    
    
  • The walk stops. isInert returns true for sap-ignore, so the pass walker never descends into it and sap neither reads nor paints inside (dom.js:14-16,25). The editor's own markup is left exactly as authored; it round-trips because Hyperclay saves the live DOM as-is.
    • 
    
  • The linter stops too. A sap-ignore subtree is skipped whole during mount analysis, so foreign markup or a third-party editor's attributes inside it never trip E01 (lint.js:33-35). (A template subtree is still scanned; only sap-ignore is fully exempt.)
    • 
    
  • The Notion shape. sap owns the block list (items/item + trigger-add/move:/trigger-remove); each block body is a sap-ignore editor mount. Add, remove, and reorder stay fully reactive; the prose stays rich.
    • 
  

  
<main sap>
  <ul items="blocks">
    <li item template>
      <input bind="kind">                <!-- block metadata: reactive -->
      <button move:up>↑</button>
      <button trigger-remove>✕</button>
      <div sap-ignore no-undo
           contenteditable="true">…</div> <!-- the real editor: sap never reads this -->
    </li>
    <!-- …authored rows… -->
  </ul>
  <form trigger-add="blocks"><input bind="kind" required></form>
</main>


  
Three caveats when you embed (all code-grounded)

  

    
1 · event noise
sap-ignore stops the DOM walk, not event delegation. sap's input/change listeners sit on document and schedule a pass for any event bubbling out of [sap], ignored subtree or not (actions.js:231-245,252-253). Firewall: stopPropagation() the editor's input/change. sap's listener is bubble-phase, so it stops; Hyperclay's [persist] is capture-phase (enablePersistentFormInputValues.js:50,64) and autosave is a MutationObserver, so neither is disturbed.

    
2 · undo / autosave
A rich editor churns the DOM. Put no-undo on the mount so its keystrokes do not flood Hyperclay's undo stack; the region stays watched, so autosave still fires (region-policy.js:75). Avoid no-watch here: it also turns off autosave-on-edit and undo for the subtree (region-policy.js:73,90), so a prose-only change would never mark the page dirty.

    
3 · not in sap state
sap cannot read the prose, so you cannot calc:/count/search over it. If you need a word count or search index, mirror a derived plaintext into a normal field with a programmatic write: Sap(el).body = editor.textContent.

    
JS-state editors
Editors that keep their canonical content in JS rather than the DOM (a TinyMCE iframe, Slate's React tree) leave the cloned DOM stale at save time. Flush content into the mount before serialize: register onSnapshot (runs on save and live-sync) or an inline [onbeforesave] (save only) (snapshot.js:55-57,132-134).

03Extending sap / plugins


  

    short answer
    
There is no plugin system, and the vocabulary is frozen. The only real extension point is custom text formats: Sap.formats.x = fn or Sap.config({ formats }). The expression-helper signature is frozen forever. It is frozen on purpose, to keep failures loud and the contract zero-ambiguity.

  


  
The one extension API (with a working example)

  
// register a custom format — either form works
Sap.formats.eur = (n) => "€" + Number(n).toFixed(2);
Sap.config({ formats: { gbp: (n) => "£" + Number(n).toFixed(2) } });

// then use it anywhere a built-in format goes
<span text:eur="state.price"></span>

  
    
    
  • Format registry: the formats object lives in helpers.js:67-91. applyFormat(name, v) looks it up and throws a tagged E22 for an unknown name, or for a numeric built-in format applied to non-finite input (helpers.js:97-114).
    • 
    
  • Registration surface: Sap.formats = formats (sap.js:129) and Sap.config({formats}) which does Object.assign(formats, options.formats) (sap.js:131-134).
    • 
    
  • Frozen signature: ["state","item","el","root","fmt","num","sum","count","avg","min","max","plural","days"] (compile.js:6). Custom formats extend the fmt namespace object passed at runtime, never the parameter list, so the source-keyed compile cache never invalidates (compile.js:26-41).
    • 
  


  
Why it is frozen (the real reason)

  

    
Loud failure / zero ambiguity
A foreign-dialect attribute (x-text, v-if, @click, :class) hard-errors with E01 and teaches the sap spelling (errors.js:69-82, lint.js:69-77). If authors could add vocabulary, an unknown attribute could not be told apart from a typo, and the did-you-mean guarantee collapses.

    
Cache stability + agent-authorability
Extending the parameter list would invalidate the source-keyed cache; the fmt namespace sidesteps that. And a finite, frozen vocabulary is exactly what makes sap files reliably writable and checkable by an LLM.

  


  

    Why there is no live "foreign dialect" demo on this page. Mounting <output x-text="…"> would intentionally produce sap ✗ E01 and a [sap-error] beacon, which would fail this page's own verification gate. So here is the engine's real console output verbatim — what block() prints via console.error (errors.js:116-121). Sap.report() returns the same facts as a JSON object instead: {ok, app, halted, errors[], warnings[]} (errors.js:137-146).
    
sap ✗ E01 foreign-dialect
  at main[sap] > output — x-text="state.total"
  problem: "x-text" is a foreign-dialect attribute; Sap has no x-text directive
  fix: use text="state.x" (Sap paints text only) — or wrap the subtree in sap-ignore if intentional

04All the ways to set state


  

    short answer
    
State is "set" by anything that changes the DOM, because every change is just the next pass's input. The deliberate write surfaces are: state= declarations, two-way bind= per control type, the Sap(el) write-through proxy and its $-verbs, the action attributes (set:, trigger-*, move:, sort:), form trigger-add, and detail= write-back. They converge on one scheduler/pass model; control-backed programmatic writes share the carrier event path, while attribute-state and row-order writes schedule directly.

05Modals / showing item details


  

    short answer
    
No. Sap has zero modal coupling: no modal attribute, no modal API. The detail="LIST by KEYEXPR" lens projects a selected row into any container — a <div>, a <form>, or a native <dialog> — and edits route back to the source row. You bring whatever modal you like; sap just fills it.

  


  
The mechanism

  
    
    
  • handleDetail parses LIST by KEYEXPR, resolves the list, evaluates the key, finds the row whose $key matches, then unhides the container (el.hidden = false) and walks it in projection mode; no match toggles hidden = true (pass.js:160-197). A malformed spec or throwing key expr just hides it, silently.
    • 
    
  • Projection paints write each bound control silently and skip the active element, so typing is never clobbered (pass.js:261-271).
    • 
    
  • Edits route back: the delegated input/change handler detects an input inside a detail panel with a resolved _sapDetailRow, finds the source row's bound control via ownedBind, and writes through it (actions.js:236-243). Sap(this) inside the panel resolves the projected row, so $remove()/$move() operate on the source row (scope.js:44-57).
    • 
    
  • Because the lens only toggles hidden and writes text/values, the container type is irrelevant. The one limit: an items= list nested inside a detail= panel is mount error E17 (lint.js:166-168).
    • 
  


  

    Timing detail, handled in the demo. set: schedules an async pass, but dialog.showModal() is synchronous, so a naive set: + inline showModal() can open the dialog one tick before the projection fills. The robust pattern (used below): one inline handler that sets state, calls Sap.refresh() for a synchronous pass, then opens. Also note: state="open" on a <dialog> is a hard error E33 (lint.js:130-132) — drive visibility with showModal(), not state.
